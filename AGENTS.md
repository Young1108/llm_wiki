# LLM Wiki - Agent 规范

本文件定义了你（LLM Agent）如何维护这个 wiki。每次会话开始时都要先阅读它。它是本知识库的最高规范，统一约束目录结构、命名方式、工作流、语言风格与维护规则。

---

## 目录结构

```text
LLM wiki/
├── AGENTS.md          ← 当前文件；Agent 的工作规范
├── README.md          ← 面向人的说明文档
├── raw/               ← 原始来源文档（只读，永不修改）
│   ├── assets/        ← 来源中引用的本地图片
│   └── *.md / *.pdf / *.txt / …
└── wiki/              ← Agent 的工作区；由 Agent 全权维护
    ├── index.md       ← 主目录；每次摄取/沉淀后都要更新
    ├── log.md         ← 追加式时间日志
    ├── overview.md    ← 全库总览与阶段性综合
    ├── sources/       ← 每个来源对应一页摘要
    ├── entities/      ← 人物、组织、产品、地点等实体页
    ├── concepts/      ← 概念、框架、方法、术语等概念页
    └── analyses/      ← 对比分析、专题研究、问答沉淀
```

---

## 命名规则

- 文件名继续使用英文小写 kebab-case，例如 `transformer-architecture.md`
- 空格改为连字符，去掉多余标点，保持简短且可识别
- 来源摘要页固定写入 `wiki/sources/<slug>.md`
- `slug` 应与 `raw/` 中的来源文件名稳定对应

---

## 语言规范

### 默认写作语言

除非用户明确要求其他语言，`wiki/` 下的新内容与更新内容一律使用简体中文，包括：

- 页面标题
- 正文叙述
- 摘要、结论、分析
- 表格说明文字
- 日志正文
- 面向用户的最终回答

### 保持英文的结构标识

为了保证结构稳定，以下内容保持英文或既有 slug，不做中文化迁移：

- 文件名与路径
- wikilink 的目标 slug，如 `[[concepts/rag]]`
- frontmatter 中的 `type`
- frontmatter 中的 `sources` 与 `related`

如需在正文中展示中文名称，优先使用带别名的链接形式，例如 `[[concepts/rag|RAG（检索增强生成）]]`。

### 处理中英文来源的规则

- 原始来源 `raw/` 保持原文，不翻译、不改写
- 来源摘要页默认用中文总结
- 原文中值得保留的关键引文可以保留原文，并在上下文中用中文解释其意义
- 专有名词首次出现时，优先使用“中文名（英文名）”或“英文名（中文说明）”

---

## 页面 Frontmatter

除 `index.md` 与 `log.md` 外，所有 wiki 页面都应以 YAML frontmatter 开头：

```yaml
---
title: "页面标题"
type: concept | entity | source | analysis | overview
tags: [标签1, 标签2]
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [source-slug-1, source-slug-2]   # 影响本页的来源
related: [page-slug-1, page-slug-2]        # 相关交叉链接
---
```

---

## Wikilink 规则

内部交叉引用使用 Obsidian 风格链接：`[[page-slug]]` 或 `[[page-slug|显示文本]]`。

- 永远链接到已存在页面的文件名 stem（不带 `.md`）
- 新建页面后，要回看 `wiki/index.md` 和相关页面，补齐应该指向它的反向链接
- 面向中文阅读时，优先使用带中文别名的链接展示

## 工作流：摄取来源

当用户说“ingest [来源]”或把文件放入 `raw/` 后：

1. **读取** 来源文件；如果有图片且能力允许，也可查看引用图片
2. **与用户确认重点**，必要时询问更关注哪个角度
3. **写入** `wiki/sources/<slug>.md`，形成结构化摘要，至少包括：
   - 这是什么（类型、作者、时间、来源）
   - 核心主张 / 关键发现
   - 值得保留的引文
   - 它提出了哪些问题
   - 它与现有 wiki 内容的关系
4. **更新** `wiki/index.md`，加入新的来源页
5. **更新** 受该来源影响的实体页与概念页（通常 3-10 页）
   - 增补新信息
   - 如与旧说法冲突，显式标出矛盾
   - 更新 frontmatter 中的 `sources`
6. **必要时更新** `wiki/overview.md`，如果该来源显著改变了整体综合判断
7. **追加** 到 `wiki/log.md`：

   ```text
   ## [YYYY-MM-DD] 摄取 | 来源标题
   - 来源文件：`raw/source-name.md`
   - 摘要页面：[[sources/slug|中文标题]]
   - 更新页面：[[entities/foo|中文名]]、[[concepts/bar|中文名]]
   - 新增判断：……
   - 待解问题：……
   ```

## 工作流：回答问题

当用户提出问题时：

1. 先读 `wiki/index.md`，定位相关页面
2. 再读相关页面，并基于这些页面综合回答
3. 回答默认使用中文，并尽量附上 wiki 内部引用，例如 `[[page|中文标题]]`
4. 如果回答本身有长期价值（如对比、专题分析、重要结论），主动询问是否要沉淀到 `wiki/analyses/<slug>.md`
5. 如果用户同意沉淀：
   - 更新 `index.md`
   - 追加 `log.md`

   ```text
   ## [YYYY-MM-DD] 问答 | 问题摘要
   - 是否沉淀：已沉淀到 [[analyses/slug|中文标题]]
   - 参考页面：[[...]]、[[...]]
   ```

## 工作流：巡检 Wiki

当用户说“lint”或“健康检查”时，执行如下检查并输出中文报告：

| 检查项 | 判定方式 |
|---|---|
| 矛盾 | 不同页面中的主张彼此冲突 |
| 过时内容 | 页面 `sources` 已被更新来源显著覆盖 |
| 孤儿页 | `wiki/` 中没有任何入链的页面 |
| 缺失页面 | 某实体/概念被多处提到却没有独立页面 |
| 缺失交叉链接 | 明显相关的两个页面没有互相链接 |
| 信息空白 | wiki 还答不上来的关键问题，可建议后续找什么来源 |

巡检结果使用 markdown checklist 输出，并询问用户是否要立即修复。

同时追加到 `wiki/log.md`：

```text
## [YYYY-MM-DD] 巡检
- 发现问题：N
- 已修复：M
- 未解决：……
```

## 内容约定

### 矛盾标注

当新信息与既有主张冲突时，使用如下格式：

```text
> ⚠️ **矛盾**（添加于 YYYY-MM-DD，来源：[[sources/slug|中文标题]]）：[新主张]。此前表述为：[旧主张]。
```

### 置信度标记

在合适时，可在结论末尾添加：

- `[高]` - 多个来源一致支持
- `[中]` - 有一个较强来源支持
- `[低]` - 推断、外推或弱来源支持
- `[?]` - 仍不确定，值得继续验证

### 永不修改 `raw/`

`raw/` 下所有文件都是原始事实来源。你只能读取，不能编辑、移动或删除。

### `index.md` 必须实时准确

`index.md` 是 Agent 的导航地图。任何页面被创建、改名、删除或新增沉淀后，都要立即更新它。

### 每次操作都要记日志

`log.md` 采用追加式记录。常规情况下不回写历史条目。统一使用：

```text
## [YYYY-MM-DD] <类型> | <标题>
```

推荐类型：

- `初始化`
- `摄取`
- `问答`
- `巡检`
- `规范更新`

标题与正文默认使用中文。

---

## 输出形式

根据用户需求，回答或沉淀可以采用以下形式：

- **Markdown 页面** - 默认形式；有长期价值时写入 `wiki/analyses/`
- **对比表** - 使用 markdown table
- **Marp 幻灯片** - 在 frontmatter 中加入 `marp: true`
- **Python / matplotlib 图表** - 生成用户可本地运行的代码

---

## 规范演进

本文件由用户与 Agent 共同维护。如果某项约定不再适用，应先提出修改建议，达成一致后再更新本文件。

规范变更也要写入 `wiki/log.md`，格式示例：

```text
## [YYYY-MM-DD] 规范更新 | 调整中文化策略
- 变更内容：wiki 正文、标题、日志与后续沉淀默认改用中文
- 保持不变：文件名、slug、frontmatter 的结构字段
```
